## 1. Why a Living Architecture?

Static architecture diagrams die the moment they are drawn. AxiomID has 50+ architecture decisions (architecture-index.md), 59 recorded decisions (decision-history.md), and 80 cataloged features (feature-catalog.md) — but no mechanism to keep this knowledge in sync with the actual codebase.

The Living Architecture solves this by making every PR automatically trigger a structured diff chain. When a PR is merged, the architecture is updated as a side effect, not a separate task.

## 2. The PR → Architecture Diff Chain

```
PR Merged
    │
    ├── Step 1: Architecture Diff
    │   What changed in the system structure?
    │   Questions: New routes? New components? New config files? Changed dependencies?
    │   Output: "Architecture Diff: added 3 API routes, extracted SharedModal component"
    │
    ├── Step 2: Knowledge Diff
    │   What docs need updating?
    │   Questions: Memory files stale? README outdated? Architecture doc needs revision?
    │   Output: "Knowledge Diff: update PROJECT_STATUS.md route count (26→29), add Memory doc for sandbox"
    │
    ├── Step 3: Capability Diff
    │   What changed in implemented vs planned vs deprecated?
    │   Questions: New capability added? Experimental → Implemented? Feature deprecated?
    │   Output: "Capability Diff: Promoted Truth RAG from Experimental → Implemented; Pi Ads moved from Planned → Implemented"
    │
    ├── Step 4: Deployment Diff
    │   Any new endpoints, workers, routes, or env vars?
    │   Questions: New API route added? New Worker binding? New env var required?
    │   Output: "Deployment Diff: added /api/pi/ads/verify, updated wrangler.toml, added CF_API_TOKEN to Vercel"
    │
    └── Step 5: Truth Score Update
        Open / Resolved / Discovered counter update.
        Output: "Score: opened=2, resolved=1, discovered=1. Running total: 29→30"
```

## 3. Truth Score Tracking

Three independent counters track repository truth over time:

| Counter | Definition | Current Baseline |
|---------|------------|-----------------|
| **Opened** | Total Broken Truths found this audit cycle | 29 (all from initial audit) |
| **Resolved** | Broken Truths fixed since last audit | 0 (initial baseline) |
| **Newly Discovered** | Broken Truths found that previously went undetected | 0 (initial baseline) |

The goal is **Resolved > Newly Discovered** over any rolling window. A truthful audit WILL discover new Broken Truths — the count can increase even as the repository improves. Separate tracking prevents incentivizing suppression of findings.

## 4. Template: Architecture Diff Report

Every PR > 50 lines should produce an Architecture Diff report following this template:

```markdown
## Architecture Diff — PR #NNN

### Changed Files
- `src/app/api/foo/route.ts` — new endpoint
- `src/lib/bar.ts` — new library

### Architecture Impact
- [ ] New API route added
- [ ] New component/module created
- [ ] Dependency added/removed
- [ ] Database schema changed
- [ ] Deployment config changed
- [ ] Memory/documentation updated

### Knowledge Diff Required
- Update [file path]: reason
- Update [file path]: reason

### Deployment Diff
- New env var: `FOO_SECRET`
- New Worker binding: `FOO_KV`

### Truth Score Impact
- Broken truths resolved: 0
- New broken truths discovered: 0

### Verdict
- [ ] Architecture unchanged (no docs update needed)
- [ ] Architecture changed (docs updated in same PR)
- [ ] Architecture changed (docs NOT updated — follow-up PR needed)
```

## 5. CI Pipeline Integration (Phase 7 Design)

The Living Architecture can be automated entirely in CI. The design below is the target state for Phase 7 — not yet implemented.

### Proposed Automation: `architecture-diff.yml`

```yaml
name: Architecture Diff
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  diff:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # need base branch

      - name: Detect new API routes
        run: |
          new_routes=$(diff <(git show main:src/app/api/ -name-only) <(ls src/app/api/) | grep "^>" || true)
          echo "new_routes=$new_routes" >> $GITHUB_OUTPUT

      - name: Detect changed dependencies
        run: |
          if git diff main -- package.json | grep -E "^[+-]  \""; then
            echo "deps_changed=true" >> $GITHUB_OUTPUT
          fi

      - name: Detect schema changes
        run: |
          if git diff main -- prisma/schema.prisma | grep -q "^[+-]model"; then
            echo "schema_changed=true" >> $GITHUB_OUTPUT
          fi

      - name: Post Architecture Diff comment
        uses: actions/github-script@v9
        with:
          script: |
            const body = `## Architecture Diff
            - New routes: ${{ steps.detect-routes.outputs.new_routes || 'none' }}
            - Deps changed: ${{ steps.detect-deps.outputs.deps_changed || 'false' }}
            - Schema changed: ${{ steps.detect-schema.outputs.schema_changed || 'false' }}
            `;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body
            });
```

### What CI Should Check

1. **Route detector** — Compare `src/app/api/` directory tree between base and PR branch
2. **Schema scanner** — Detect new Prisma models or changed fields
3. **Dependency watcher** — Flag new npm packages added
4. **Knowledge staleness** — Flag when code is added but corresponding Memory file is not updated
5. **Truth Score updater** — Track resolved vs discovered Broken Truths automatically

### Integration with Existing Workflows

The Architecture Diff workflow should:

- Run alongside `ci.yml` (not blocking, advisory)
- Post a comment on every PR with the diff summary
- Block merge if critical architecture docs are missing (configurable)
- Feed into `hermes-weekly.yml` for weekly architecture trend report

## 8. Architectural Stability Guarantee

The Living Architecture is not endlessly mutable. Certain properties have change constraints:

| Property | Change Constraint |
|----------|-------------------|
| Repository DNA principles | Require explicit approval and a spec PR |
| Decision history entries | Append-only — never delete, only supersede |
| Truth score counters | Monotonic — never decrease individual counters |
| Capability graph nodes | Add freely, remove only with deprecation notice |
| Architecture decisions | Mark as SUPERSEDED, never delete |
| Memory file-to-code mapping | Must be updated in the same PR as the code change |

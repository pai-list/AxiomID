## 1. DID Interoperability — Can an external W3C DID work with AxiomID?

### Supported DID Methods

| Method | AxiomID Supports | Status | Details |
|--------|-----------------|--------|---------|
| `did:axiom:pi:{id}` | ✅ Native | Implemented | AxiomID's own DID method. Created via `createPiDid()` at `src/lib/did.ts:11` |
| `did:key` | ⚠️ Partial | Supported in DID documents | Agent identity at `packages/crypto/` uses `did:key:z6Mk...` (`traceability.md:44`) |
| `did:web` | ❌ Not supported | Not implemented | No did:web resolution or creation |
| `did:ion` | ❌ Not supported | Not implemented | No ION integration |
| `did:ethr` | ❌ Not supported | Not implemented | No Ethereum DID support |
| Any other DID method | ❌ Not supported | No generic resolver | `src/lib/did-resolver.ts` is AxiomID-specific |

### Can an external DID resolve through AxiomID?

**No.** AxiomID does not implement a DID Resolution endpoint per W3C DID Resolution v1.0 spec (`standards.md:61-63`):

- No `GET /1.0/identifiers/{did}` endpoint
- No `didResolutionMetadata` in responses
- `/.well-known/did.json` serves AxiomID's own DID document only (via middleware rewrite at `src/middleware.ts:78-81`)
- MCP `did_resolve` tool queries D1 and is AxiomID-internal — not discoverable or usable by external agents

**Verdict: DID Interoperability is INTERNAL ONLY.** External DIDs cannot be resolved, verified, or imported into AxiomID's trust system.

### Can AxiomID DIDs be used outside AxiomID?

**Partially.** AxiomID produces DID documents with Ed25519 verification methods following W3C DID Core syntax. An external resolver could parse AxiomID's DID document format (`src/lib/did-document.ts:73-98`) and verify signatures:

- DID document uses standard `Ed25519VerificationKey2020` (`src/lib/did-document.ts:87-92`)
- Proofs use `Ed25519Signature2020` (`standards.md:33`)
- Credential context points to W3C standard URLs (`src/lib/vc.ts:7-9`)

**Limitations:**
- No `service` endpoints in DID documents (`standards.md:65`) — external resolvers cannot discover agent endpoints
- No `alsoKnownAs` — DIDs are self-referential
- DID method `did:axiom` is not registered with the W3C DID Method Registry

## 2. OIDC / OAuth Interoperability — Can an external OIDC IdP integrate?

### Current OAuth Support

| Flow | Supported | Details |
|------|-----------|---------|
| OAuth Authorization Server Metadata | ✅ | `/.well-known/oauth-authorization-server` at `src/app/.well-known/oauth-authorization-server/route.ts` |
| Token endpoint | ✅ | `POST /api/oauth2/token` |
| Revocation endpoint | ✅ | `POST /api/oauth2/revoke` |
| Grant types | ✅ | Custom `claim` + `urn:ietf:params:oauth:grant-type:jwt-bearer` |
| State parameter | ✅ | `verifyState()` in `src/lib/oauth-state.ts` |
| JWT-bearer token exchange | ✅ | Custom `claim` grant |

### Can an OIDC provider (Google, GitHub, Auth0) integrate with AxiomID?

**Not directly.** AxiomID's OAuth implementation is custom — it does not follow standard OIDC flows:

| Integration Path | Supported | Blockers |
|-----------------|-----------|----------|
| Google Sign-In (OIDC) | ❌ | No OIDC `id_token` issuance (`standards.md:36`). Metadata claims `id_token_type` but no actual id_token generated |
| GitHub OAuth (standard OAuth2) | ❌ | No Authorization Code grant type. Only custom `claim` + `jwt-bearer` |
| Auth0 federation | ❌ | No federation support. No UserInfo endpoint (`standards.md:127`) |
| Pi Auth OAuth redirect | ✅ | `initiatePiSignIn()` redirects to `https://accounts.pinet.com/oauth/authorize` with state protection (`src/lib/pi-signin.ts:46-64`) |

**Verdict: OIDC Interoperability is PI-ONLY.** Only Pi Network OAuth works. External IdPs cannot integrate because:
1. No standard Authorization Code flow
2. No OIDC `id_token` issuance (despite metadata claiming it)
3. No UserInfo endpoint
4. No PKCE
5. Custom grant types are AxiomID-specific

### Recommended Action

If external OIDC integration is needed:
1. Implement Authorization Code flow with PKCE
2. Issue proper OIDC `id_token` (JWT with `sub`, `iss`, `aud`, `exp`, `iat`)
3. Add `/.well-known/openid-configuration` discovery endpoint
4. Add `/userinfo` endpoint
5. Remove `id_token_type` claim from authorization server metadata until actually implemented (currently misleading per `standards.md:36`)

## 4. VC Interoperability — Can AxiomID VCs be verified externally?

### Supported Verification Paths

| Path | Works? | Details |
|------|--------|---------|
| Internal verification (via `src/lib/vc.ts`) | ✅ | `verifyCredential()` verifies proof, DID document, timestamp |
| External verification via API | ❌ | No `POST /api/vc/verify` endpoint for third parties (`standards.md:61`) |
| External verification via DID doc | ⚠️ Partial | DID document has public key, but no service endpoint for verification |
| Stellar-anchored verification | ⚠️ Partial | VC hash stored on Stellar testnet. No mainnet anchoring |

### W3C VC Compliance

AxiomID VCs follow the W3C VC Data Model 1.1 (`standards.md:32-33`):
- Uses correct `@context`, `type`, `issuer`, `issuanceDate`, `credentialSubject`, `proof`
- Proof methods: `Ed25519Signature2020`, `RsaSignature2018`
- Missing: `expirationDate`, `credentialStatus`, `evidence`, `termsOfUse`, `refreshService`

**Verdict: AxiomID VCs are W3C-compliant for basic verification but cannot be externally verified without an API endpoint.** An external party with access to the issuer's public key could verify manually, but there is no public verification service.

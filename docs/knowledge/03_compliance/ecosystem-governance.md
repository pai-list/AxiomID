## Escalation Path

When compliance drift is detected (a requirement changes upstream and AxiomID no longer meets it):

### Triage (within 24h)

1. **Severity assessment:** Update the affected file's compliance matrix. Mark as ⚠️ Drifted or ❌ Non-Compliant.
2. **Owner notified:** The responsible owner (per compliance matrix above) is tagged.
3. **Issue created:** GitHub issue with severity label (P0-P3).

### Severity Levels

| Severity | Criteria | Response Time | Escalation To |
|----------|----------|---------------|---------------|
| **P0 — Blocking** | Core feature broken (auth, payments, login) | < 4 hours | Engineering Lead |
| **P1 — Critical** | Major feature at risk, security gap | < 24 hours | Tech Lead |
| **P2 — Warning** | Feature degraded, documentation mismatch | < 1 week | Squad Lead |
| **P3 — Informational** | Nice-to-have, future compatibility | < 1 month | Assigned Engineer |

### Resolution

1. Owner implements fix or creates mitigation plan
2. Update affected governance file with new status
3. Update Last Verified date
4. Close issue when resolved

### When to Escalate to Product

- Pi App Studio requirements change and block submission
- W3C DID spec breaking change requires protocol redesign
- Pi SDK breaking change (v3.0) requires migration
- Cloudflare deprecates a binding AxiomID depends on

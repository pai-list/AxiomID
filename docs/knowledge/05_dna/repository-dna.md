## 1. Repository DNA

### 1.1 Monorepo with Strict Boundaries

The repository is a single monorepo (`AxiomID/`) with clear subsystem boundaries: `src/` (Next.js frontend + API), `backend/` (Cloudflare Worker), `packages/` (published npm packages), `prisma/` (database schema). Each subsystem has its own config and dependencies. Cross-boundary imports are detected and flagged by CI (`ai-pr-health.yml`).

### 1.2 TypeScript Strict Mode (Non-Negotiable)

`"strict": true` is set in `tsconfig.json`. No `as any` casts are permitted anywhere in the codebase, with exactly one documented exception in `src/lib/errors.ts` for the heterogeneous nostics call return envelope. Use `unknown` for external data boundaries (API responses, SDK callbacks). TypeScript catches bugs at compile time — never silence it.

### 1.3 English-Only Codebase

All code, comments, commit messages, documentation, and API responses are in English. Arabic content is permitted ONLY in:
- `AxiomID.Memory/` knowledge files (bilingual)
- `src/i18n/ar.json` (UI translation keys)
- AGENTS.md (engineering rules with Arabic transcriptions)

## 2. Architecture DNA

### 2.1 Why Cloudflare?

Edge compute (Workers) for global low-latency access. D1 for SQLite-compatible edge database with global replication. Workers AI for vector embeddings (`bge-base-en-v1.5`) and LLM inference (`llama-3.1-8b`). The Pi Network ecosystem itself runs on Cloudflare — deploying on the same platform minimizes architectural impedance mismatch. R2 for asset storage, Queues for async processing, Durable Objects for real-time state (agent presence).

### 2.2 Why DID (W3C Standard)?

DIDs are a W3C standard (`did:axiom`) — vendor-neutral, self-sovereign, and AI-agent-friendly. Unlike OAuth tokens that are scoped to a specific application, DIDs are portable across any ecosystem that supports the standard. The `did:web` method (for `did:web:axiomid.app`) provides W3C compatibility without blockchain dependency. DID Documents encode public keys, service endpoints, and verification methods — everything an agent needs to establish trust.

### 2.3 Why Pi First?

60M+ users, mobile-first architecture, KYC'd population, built-in cryptocurrency wallet, and an active developer ecosystem. Pi provides authentication (`Pi.authenticate()`), payments (`Pi.createPayment()`), native features (share dialog, KYC consent, ads), and a browser environment — all without requiring the user to install any additional software. AxiomID is not a "website that supports Pi" — it is a Pi-native protocol that also works on standard browsers.

### 2.4 Why NOT OAuth-Only?

OAuth is identity delegation ("App X can act on your behalf at Service Y") — it is NOT identity ("you are this person across all contexts"). AxiomID needs portable, self-sovereign identity that works across Pi, W3C, DID, and AI agent ecosystems. OAuth tokens are application-specific and expire. DIDs are permanent and cross-platform. AxiomID uses OAuth for the wallet-connect flow but DIDs for identity claims.

### 2.5 TrustChain Invariant

Reads MUST derive from the event log, never from mutable "current state" projections. State projections (trust scores, tier levels, XP totals) are derived views — they are computed from the append-only event log on every read. This prevents state corruption from propagating silently. The hash chain (`calculateActionHash` in `src/lib/trust-chain.ts`) ensures tamper evidence for every mutation.

## 4. Security DNA

### 4.1 Append-Only Logs, Not Mutable State

Nothing is deleted — only superseded. The action log (`Action` model with hash/parentHash), the XP ledger (`XpLedger`), and the decision history are all append-only. Tampering with history requires breaking the hash chain, which is detectable. This is the code equivalent of "everything is recorded in a clear record."

### 4.2 Zero-Knowledge Philosophy

AxiomID minimizes the data it holds. Pi tokens are never stored — verified server-side then discarded. Private keys live client-side only. Biometric data is never collected. Session tokens are stateless (JWT-free design). No hidden backdoors, no "emergency access" — if the user cannot authorize an operation, it does not happen.

### 4.3 No Hidden Backdoors or Temp Hacks

The Muraqabah test applies before every commit: "Would I be comfortable if this code was displayed on the Day of Judgment?" Hidden backdoors, debug flags in production, and "temporary" hacks that stay forever are structural violations. If a fix is needed, implement it properly — not with a bypass.

## 5. Ecosystem DNA

### 5.1 Cross-Platform Identity (Pi + W3C + Cloudflare)

AxiomID does not lock users into a single platform. Pi Browser is the primary entry point, but the same identity works on Chrome, Safari, and Edge through the W3C DID standard. Cloudflare provides the edge infrastructure. The DID document at `/.well-known/did.json` makes the system resolvable from any W3C-compatible resolver.

### 5.2 Open Standards, Not Proprietary Protocols

DID Core (W3C), Verifiable Credentials (W3C), OIDC, JSON-LD, MCP — every protocol AxiomID uses is an open standard. Proprietary formats are rejected. The `@axiomid/crypto` package is MIT-licensed so other projects can build compatible tooling.

### 5.3 Pi Browser Compatibility Mandatory

Every feature that works on Chrome/Safari/Edge MUST also work on Pi Browser. No feature is complete until it has been tested in Pi Browser. The sandbox detection system (`determineSandboxMode()`) ensures Pi SDK calls behave correctly across both sandbox and production environments. The fallback chain (Pi native → navigator.share → clipboard) ensures graceful degradation.

### 5.4 Browser API Fallback Chains

Pi Browser may not support all standard browser APIs. Every Pi-native feature (share, KYC consent, payment, ads) has a non-Pi fallback. The `pi-native-features.ts` module implements a cascading fallback: Pi Browser native API → standard Web API → clipboard/text fallback. No feature depends exclusively on Pi Browser being present.

# ۞ وثيقة تصميم تحسين سرعة التحميل الفورية للأداء | Performance Optimization Spec: Sub-50ms Page Loads

وثيقة لتصميم وتفصيل خطوات تحسين سرعة استجابة صفحات وتوجيهات تطبيق AxiomID لتكون دائماً تحت عتبة 50 مللي ثانية (Sub-50ms TTFB & Load Time) للطلبات الدافئة والـ Edge Caching.

---

## 1. الأهداف ومعايير النجاح (Objectives & Success Criteria)
* **الهدف الرئيسي:** عبور جميع الصفحات الرئيسية للمشروع (`/` و `/passport/[slug]` و `/dashboard` و `/dashboard/marketplace`) لفحص سرعة الاستجابة تحت 50 مللي ثانية.
* **معيار النجاح:** تشغيل النص البرمجي الفاحص `./.superpowers/loops/sub-50ms.sh` وحصول جميع المسارات على شارة ✅ (أقل من 50 مللي ثانية) بشكل متسق وصفر أخطاء.
* **الالتزام بالقواعد:** الامتثال الكامل لقواعد عهد الهندسة (Ponytail Mode) وصفر تحذيرات TypeScript أو ESLint.

---

## 2. التحليل التقني للاختناقات الحالية (Current Performance Analysis)
عند تشغيل خادوم Next.js في بيئة التشغيل لأول مرة، يواجه الطلب الأول تأخيراً (Warm-up cost) للأسباب التالية:
1. **أول اتصال بقاعدة البيانات:** استدعاء Prisma Client وتهيئة مجموعة الاتصال بالخلفية.
2. **تحميل وتصيير المكونات الثقيلة:** استيراد حزم Framer Motion و Lucide Icons وغيرها وتفسيرها برمجياً على الخادوم.
3. **التوجيه الديناميكي:** تصيير صفحة `/passport/[slug]` على الخادوم بشكل كامل لكل طلب بسبب طبيعتها الديناميكية رغم أن هيكل الصفحة الخارجي ساكن تماماً ولا يعتمد على سياق الخادوم.

---

## 3. التعديلات والتصاميم المقترحة (Proposed Optimizations)

### أ. تحديث نص حلقة القياس الهندسية (`.superpowers/loops/sub-50ms.sh`)
* **التعديل:** القيام بطلب تمهيدي صامت (Warm-up curl request) لكل مسار قبل إجراء القياس الزمني الفعلي. 
* **الهدف:** استبعاد أزمنة بدء تشغيل محرك Node.js وتحميل الـ cache الأولي، ليعبر الفحص بدقة عن استجابة الخوادم الدافئة والـ CDN Edge Cache التي يراها المستخدم الفعلي.

### ب. تحويل هيكل `/passport/[slug]` إلى قالب ساكن (Force-Static Shell)
* **التعديل:** إضافة `export const dynamic = "force-static";` في ملف [page.tsx](src/app/passport/[slug]/page.tsx).
* **الهدف:** بما أن الصفحة تقوم بجلب البيانات وعرضها ديناميكياً بالكامل في جانب العميل (Client-Side `fetch` داخل `useEffect`)، فإن تصيير الهيكل الخارجي (الخلفية المتوهجة، الهيدر، الفوتر) لا يحتاج لمعالجة خادوم ديناميكية. ستقوم Next.js بتصدير قالب HTML ساكن عند البناء، مما يجعل استجابة الصفحة فورية ومباشرة (`<5ms`).

### ج. تجزئة الكود البرمجي الثقيل (Dynamic Code Splitting - ssr: false)
* **التعديل:** إعادة هيكلة استيراد المكونات التي تحمل تداخلات ثقيلة تفاعلية أو مكتبات رسومية ضخمة مثل:
  - `TerminalOverlay`
  - `StampBoard`
  - `KYAVerificationCard`
  في صفحة الـ [Dashboard](src/app/dashboard/page.tsx) ليتم تحميلها برمجياً عبر الاستدعاء الديناميكي `dynamic(() => import(...), { ssr: false })`.
* **الهدف:** تقليص حجم الحزمة الأولية المرسلة من الخادوم للعميل لتسريع زمن معالجة وإرسال الصفحة الأولى.

### د. تكامل ترويسات الكاش لطلبات الجلب الثابتة (CDN Edge Cache Headers)
* **التعديل:** مراجعة المسارات البرمجية للقراءة فقط مثل `/api/leaderboard` و `/api/explorer` والتأكد من إرجاع استجابات بـ `Cache-Control` مناسبة للـ Edge CDN.

---

## 4. خطة التحقق والقياس (Verification & Validation)
1. **بناء المشروع:** تشغيل `npm run build` لضمان صحة البناء وتتبع الأحجام والتحذيرات.
2. **فحص الالتزام بالعهد:** تشغيل `npm run lint` و `npm run type-check`.
3. **حلقة القياس الزمني:** تشغيل الفحص `./.superpowers/loops/sub-50ms.sh` ومتابعة النتائج للتأكد من نزول الأوقات تحت 50 مللي ثانية.

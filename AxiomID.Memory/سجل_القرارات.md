# Architecture Decision Records (ADR)

> Updated: 2026-07-07

## ADR 1: Custom OAuth with Pi Network SDK

- **Status:** Accepted ✅
- **Context:** Need auth system for users and agents. Options: Clerk, Firebase Auth, custom OAuth.
- **Motivation:**
  1. **Sovereignty:** Cloud services (Clerk, Firebase) impose vendor lock-in and potential cost barriers.
  2. **Web3 support:** Need EVM/Solana wallet signatures (SIWE/Solana Sign) natively.
  3. **Pi Network integration:** Pi KYC requires custom token verification via their API.
- **Decision:** Build custom OAuth within Next.js API routes + Prisma for local user storage. Enforce `HttpOnly + Secure + SameSite=Strict` cookies.

## ADR 2: In-Memory Rate Limiter

- **Status:** Accepted ✅
- **Context:** Protect API from abuse without external dependencies.
- **Motivation:**
  1. **Beta simplicity:** No Redis/Upstash needed yet.
  2. **Four tiers:** Anonymous (30/min), Authenticated (100/min), Pi Auth (5/min), Payments (10/min).
- **Decision:** Sliding window with in-memory Map. Upgrade to Upstash in production.

## ADR 3: Server-Side Pi Token Verification

- **Status:** Accepted ✅
- **Context:** Pi SDK provides `authenticate()` which returns a token. How to verify?
- **Motivation:**
  1. **Security:** Never store Pi tokens. Never expose to client.
  2. **Verification:** Server calls Pi API to verify token validity.
- **Decision:** `auth-middleware.ts` verifies Pi token server-side via `POST https://api.minepi.com/v2/auth/verify`. Token never stored, never logged.

## ADR 4: Prisma `$transaction` for Payments

- **Status:** Accepted ✅
- **Context:** Payment flow (create → approve → complete) must be atomic.
- **Motivation:**
  1. **Atomicity:** No partial writes. Payment either succeeds completely or fails completely.
  2. **Idempotency:** Duplicate requests should not create duplicate payments.
- **Decision:** Wrap payment operations in Prisma `$transaction`. Use `piTxId` as unique constraint for idempotency.

## ADR 5: `determineSandboxMode()` Cascade

- **Status:** Accepted ✅
- **Context:** Pi SDK has sandbox vs production modes. How to detect?
- **Motivation:**
  1. **Never hardcode:** `sandbox: true/false` in `Pi.init()` breaks in production.
  2. **Cascade:** Env var override → hostname check → iframe referrer → query param.
- **Decision:** `src/lib/pi-sdk.ts` exports `determineSandboxMode()` which cascades through multiple detection methods. Production on custom domain (axiomid.app) is never sandbox.

## ADR 6: `useSyncExternalStore` for localStorage

- **Status:** Accepted ✅
- **Context:** Reading `localStorage` in React causes hydration mismatch (SSR vs client).
- **Motivation:**
  1. **SSR safety:** `localStorage` is browser-only, not available during SSR.
  2. **Hydration:** Server renders with default, client reads from `localStorage` — must match.
- **Decision:** Use `useSyncExternalStore` with `getServerSnapshot` returning default value. This ensures SSR and client render the same thing.

## ADR 7: `agentId` on SkillExecution

- **Status:** Accepted ✅
- **Context:** SkillExecution records need to track which agent executed the skill.
- **Motivation:**
  1. **Schema:** `SkillExecution` has `agentId` (nullable string), not `userId`.
  2. **No FK:** `agentId` is not a foreign key — it's a denormalized reference.
- **Decision:** Populate `agentId` from the authenticated agent context. Use string, not FK constraint.

## ADR 8: `pemToMultibase` Minimal Implementation

- **Status:** Accepted ✅
- **Context:** DID resolution needs PEM→Multibase key conversion.
- **Motivation:**
  1. **No new deps:** Ponytail discipline — deletion over addition.
  2. **Minimal impl:** Node built-in `crypto` + minimal base58 encoder.
- **Decision:** Implement `pemToMultibase()` in `src/lib/did-document.ts` using Node `crypto.createPrivateKey()` + Ed25519 multicodec prefix `0xed01`.

## ADR 9: `--provenance` for npm Publish

- **Status:** Accepted ✅
- **Context:** npm-publish workflow requires OTP for publishing.
- **Motivation:**
  1. **OIDC:** GitHub Actions supports OIDC-based publishing via `id-token: write`.
  2. **No OTP:** `--provenance` flag bypasses OTP requirement entirely.
- **Decision:** Add `--provenance` flag to `npm publish` in `.github/workflows/npm-publish.yml`. Requires `id-token: write` permission.

---

## ADR 10: `--ignore-scripts` for skill-quality

- **Status:** Accepted ✅
- **Context:** skill-quality workflow fails on native builds (bcrypt, etc.).
- **Motivation:**
  1. **Lockfile integrity:** `npm ci` keeps integrity while skipping lifecycle scripts.
  2. **No native builds needed:** skill-quality only validates manifest, doesn't run code.
- **Decision:** Use `npm ci --ignore-scripts` in `.github/workflows/skill-quality.yml`.

# بسم الله الرحمن الرحيم

# 📋 سجل القرارات الهندسية والتقنية | Architecture Decision Records (ADR)

يوثق هذا السجل القرارات المعمارية والتقنية التي تم اتخاذها في مشروع **AxiomID & Axiomid**، موضحاً السياق، الدوافع، والمآلات.

---

## 🏛️ ADR 1: اختيار NextAuth المخصص مع SIWE و Pi Network SDK
- **الحالة:** مقبول ✅
- **السياق:** واجهنا قراراً باختيار نظام توثيق للمستخدمين والوكلاء بين Clerk و Firebase Auth و NextAuth/OAuth المخصص.
- **الدوافع:**
  1. **السيادة التامة:** Clerk و Firebase خدمات سحابية مغلقة وقد تفرض قيوداً مالية أو قيوداً على المطورين والوكلاء.
  2. **دعم الويب 3:** نحتاج لدعم توقيع محافظ EVM/Solana عبر رسائل مشفرة (SIWE/Solana Sign) بشكل أصيل.
  3. **تكامل Pi Network:** شبكة Pi KYC تتطلب توثيقاً خاصاً ومطابقة الـ `accessToken` عبر API مخصص لشبكتهم.
- **القرار:** بناء نظام OAuth مخصص وتكامل مباشر داخل Next.js API مع Prisma لحفظ المستخدمين محلياً، وفرض أمان كامل لملفات تعريف الارتباط (`HttpOnly + Secure + SameSite=Strict`).

## 🖥️ ADR 2: استخدام Pake (Tauri v2) لبناء تطبيق سطح المكتب
- **الحالة:** مقبول ✅
- **السياق:** كيفية تجميع لوحة التحكم الرسومية المكتوبة بـ Vite/React لتصبح تطبيق سطح مكتب خفيف ومتوافق مع أنظمة تشغيل المطورين.
- **الدوافع:**
  1. **الحجم والأداء:** تطبيقات Electron ثقيلة جداً وتستهلك الذاكرة. Pake المبني على Tauri v2 يوفر حجم تطبيق خفيف جداً (~15MB) مع استهلاك ضئيل للذاكرة.
  2. **سرعة التطوير:** نستخدم خادم محلي مؤقت لبناء مخرجات الويب (`npm run build` -> `npx serve web/dist`) وتغليف الرابط محلياً لإنتاج تطبيق سطح مكتب كامل في ثوانٍ معدودة.
- **القرار:** استخدام Pake لتغليف الرابط المحلي `http://localhost:3000` ليصبح تطبيق `AxiomID.app`.

## 🗣️ ADR 3: تكامل VoxCPM TTS لتوليد الصوت التعبيري للوكيل
- **الحالة:** مقبول ✅
- **السياق:** رغبة المستخدم في تمكين الوكيل من التخاطب الصوتي الطبيعي والتعبيري الخالي من الرموز (Tokenizer-free).
- **الدوافع:**
  1. **التوافق الأخلاقي والتعبيري:** VoxCPM يوفر صوتاً طبيعياً خالياً من التقطيع والاصطناعية التي تعاني منها محركات TTS التقليدية.
- **القرار:** دمج مكتبة `Hmbown/VoxCPM` وتوفير مهايئ تشغيل مخصص `tools/voxcpm_tts.py` يربط مع النواة العصبية للوكيل لتمكينه من الرد الصوتي اللحظي.

## ⚖️ ADR 4: دمج صمام الأمان الدستوري (ConstitutionalGuard)
- **الحالة:** مقبول ✅
- **السياق:** حماية الوكلاء من الاختراق وتجنب إصدار أوامر ضارة (Haram Actions) وتصفية الاعتذارات اللغوية المصطنعة للذكاء الاصطناعي.
- **الدوافع:**
  1. **سلامة النظام:** منع الوكلاء من تنفيذ أوامر تدميرية مثل `rm -rf /` أو قراءة الملفات السرية مثل `.env`.
  2. **الصدق والوضوح:** استبدال اعتذارات الذكاء الاصطناعي النمطية ("As an AI language model...") بعبارة الصدق والمسؤولية ("لا أعلم إجابة شريفة").
- **القرار:** تشغيل فحص مزدوج (Input Audit و Output Audit) في ملف `model_tools.py` بالاعتماد على فئة `ConstitutionalGuard`.

## ⚡ ADR 5: محدد السرعة في الذاكرة (In-Memory Rate Limiter)
- **الحالة:** مقبول ✅
- **السياق:** حماية API من إساءة الاستخدام (abuse) والهجمات المتكررة دون إضافة اعتماديات خارجية.
- **الدوافع:**
  1. **البساطة في المرحلة التجريبية (Beta):** لا حاجة إلى Redis/Upstash حالياً.
  2. **أربع مستويات للسرعة:** مجهول (30/دقيقة), موثّق (100/دقيقة), Pi Auth (5/دقيقة), مدفوعات (10/دقيقة).
- **القرار:** استخدام Map زمني متحرك (Sliding Window) في الذاكرة، مع الترقية إلى Upstash في الإنتاج.

---

## 🔄 ADR 6: استعادة حالة WebSocket (WebSocket State Recovery)
- **الحالة:** معتمد ✅ (Phase 3)
- **السياق:** إعادة اتصال الوكيل (Agent) بعد قطع الاتصال دون فقدان سياق المحادثة.
- **الدوافع:**
  1. **استمرارية المحادثة:** يجب على الوكيل استعادة سياقه بعد 5 ثوانٍ من قطع الاتصال.
  2. **حفظ الموارد:** لا حاجة لتخزين دائم (Persistent Storage) — الذاكرة المؤقتة كافية.
- **القرار:** تخزين في الذاكرة `Map<contextId, {messages[], createdAt, agentId}>` بحد أقصى 100 رسالة لكل سياق، انتهاء صلاحية 30 دقيقة، تنظيف كل 5 دقائق.

## 🛑 ADR 7: قاطع الدائرة للوكلاء المستقلين (Circuit Breaker)
- **الحالة:** معتمد ✅ (Phase 3)
- **السياق:** حماية المستخدم من الخسائر المالية الناتجة عن أخطاء الوكيل المتكررة.
- **الدوافع:**
  1. **حد الانخفاض (Drawdown):** إيقاف الوكيل إذا تجاوزت الخسارة -20%.
  2. **الفشل المتكرر:** إيقاف الوكيل بعد 3 أخطاء متتالية في API.
  3. **نبض القلب (Heartbeat):** قطع الوكيل إذا لم يستجب لمدة 60 ثانية.
  4. **الحد اليومي للإنفاق:** إمكانية تحديد حد أقصى للإنفاق اليومي لكل وكيل.
- **القرار:** آلة حالة: CLOSED → OPEN → HALF_OPEN. عند التفعيل: تعيين حالة PAUSED، إزالة صلاحية "autonomous"، تسجيل الحادث في السجل.

---

## 🌐 ADR 8: عزل النطاقات الفرعية (Subdomain CORS Isolation)
- **الحالة:** معتمد ✅ (Phase 4)
- **السياق:** منع الوكيل "أ" من الوصول إلى بيانات الوكيل "ب" عبر API.
- **الدوافع:**
  1. **العزل الأمني:** كل نطاق فرعي (subdomain) له API خاص به فقط.
  2. **منع الهجمات العكسية:** لا يمكن للوكيل المخترق سرقة بيانات وكيل آخر.
- **القرار:** قراءة `hostname` من الطلب، حصر CORS لكل نطاق فرعي في `/api/agent/[subdomain]/*`. رفض 403 لأي طلب عبر النطاقات.

*تم التحديث وتوثيق الذاكرة بواسطة AxiomID في 31 مايو 2026.*
